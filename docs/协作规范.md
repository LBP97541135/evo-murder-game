# 进化酒馆 · 协作规范

> 本文档定义多人协作的规则、流程和约定，确保团队高效协同。

---

## 一、Git 协作规范

### 1.1 分支策略

```
main              ← 生产分支，只接受 PR 合入
  │
  ├── dev         ← 开发主分支，日常开发在此
  │   │
  │   ├── feat/xxx  ← 功能分支（如 feat/agent-registration）
  │   ├── fix/xxx   ← 修复分支（如 fix/safeactor-leak）
  │   └── docs/xxx  ← 文档分支（如 docs/api-spec）
```

**规则**：
- `main` 分支禁止直接 push，只通过 PR 合入
- 功能分支从 `dev` 创建，完成后 PR 回 `dev`
- PR 必须至少一人 review 才能合入
- 分支命名：`feat/` `fix/` `docs/` `refactor/` 前缀 + 简短描述

### 1.2 提交信息格式

```
[模块] 简述

# 模块缩写：
#   api     → 后端API
#   web     → 前端页面/组件
#   evomap  → EvoMap集成
#   agent   → Agent逻辑
#   llm     → LLM管道
#   docs    → 文档
#   infra   → 基础设施（CI/部署）
```

**示例**：
- `[api] 新增 Session 创建接口`
- `[web] 实现 Agent 配置面板`
- `[evomap] 修复 heartbeat 认证问题`
- `[agent] 实现 constitution 改写逻辑`

### 1.3 .gitignore 核心规则

```
# ⚠️ 红线：绝不提交以下内容到 git
.env                    # 包含 API Key 和 node_secret
*.db                    # SQLite 数据库文件
node_modules/           # 前端依赖
__pycache__/            # Python 缓存
build/                  # 前端构建产物
```

## 二、代码风格

### 2.1 Python（后端）

- 遵循 **PEP 8**
- 所有模块有顶层 docstring，说明模块职责
- 所有公开函数有 docstring
- 类型注解：函数参数和返回值必须标注类型
- 文件组织：每个文件一个核心职责，不超过 500 行
- 导入顺序：stdlib → 第三方 → 本项目内部

### 2.2 TypeScript（前端）

- 遵循项目内已有风格（参考已有文件）
- 所有组件文件以 PascalCase 命名（如 `AgentPanel.tsx`）
- 所有类型定义集中放在 `src/types/`
- API 调用集中放在 `src/api/`
- 状态管理集中放在 `src/providers/`
- 组件注释：每个组件顶部一行说明其用途

### 2.3 命名约定

| 类型 | Python | TypeScript |
|------|--------|-----------|
| 文件名 | snake_case | PascalCase |
| 函数/方法 | snake_case | camelCase |
| 类 | PascalCase | PascalCase |
| 常量 | UPPER_SNAKE | UPPER_SNAKE |
| 变量 | snake_case | camelCase |
| API 端点 | kebab-case | - |

## 三、开发流程

### 3.1 任务分配

- 使用 **GitHub Issues** 跟踪所有任务
- Issue 模板：`[模块] 功能描述` + 优先级标签（P0/P1/P2）
- 每个 Issue 关联对应的分支
- 完成后 Issue → PR → Review → 合入

### 3.2 优先级定义

| 优先级 | 含义 | 示例 |
|--------|------|------|
| **P0** | 核心功能，不做就无法运行 | Agent 注册、三层LLM管道、Session创建 |
| **P1** | 重要功能，Demo必须有 | 游戏界面、进化时间线UI、Memory对接 |
| **P2** | 增强功能，锦上添花 | Council界面、社区资产搜索、剧本推荐 |

### 3.3 开发顺序建议

```
第一阶段（P0 - 核心骨架）
  1. [api] 完善 evomap_client.py（实际 API 调用测试）
  2. [api] Agent 注册流程跑通
  3. [api] 三层LLM管道跑通（initial→critique→refine）
  4. [web] Agent 配置面板

第二阶段（P1 - 游戏核心）
  5. [api] Session 创建与消息传递
  6. [web] 游戏主界面（角色对话）
  7. [api] Memory record/recall 对接
  8. [web] 进化时间线UI

第三阶段（P2 - 增强体验）
  9. [api] Council 审议流程
  10. [web] 剧本库完整功能
  11. [api] Gene/Capsule 发布
  12. [web] 游戏复盘界面
```

### 3.4 测试规范

- 后端：每个 API 端点至少一个手动测试（curl 或 Postman）
- 前端：关键交互流程手动验证
- EvoMap 集成：所有 A2A 调用必须验证返回格式
- 三层管道：验证 critique 能检测到违规，refine 能修正

| 红线 | 说明 |
|------|------|
| ❌ 不提交 `.env` | 包含 API Key、node_secret |

## 七、EvoMap A2A 协议使用规范

### 7.1 认证三套体系

| 用途 | 认证方式 | 用在哪里 |
|------|---------|---------|
| Agent间通信 | `Authorization: Bearer <node_secret>` | 所有 /a2a/* 端点 |
| LLM推理 | `Authorization: Bearer sk-evomap-<API_KEY>` | /v1/chat/completions |
| 知识图谱 | `Authorization: Bearer ek_<API_KEY>` | /kg/* 端点（可选） |

### 7.2 GEP-A2A 信封格式

所有 A2A 端点请求必须包含协议信封：

```json
{
  "protocol": "gep-a2a",
  "protocol_version": "1.0.0",
  "message_type": "<hello|publish|heartbeat|...>",
  "message_id": "msg_<timestamp>_<random>",
  "sender_id": "<node_id>",
  "timestamp": "<ISO8601>",
  "payload": { ... }
}
```

### 7.3 积分管理

- 每个节点注册获 100 启动积分
- Memory recall 消耗 2 积分
- Memory record 消耗 1 积分
- 资产获取消耗 0-12 积分（按 GDI 分层）
- 发布（publish）免费
- 注意积分余额，避免 Agent 离线
