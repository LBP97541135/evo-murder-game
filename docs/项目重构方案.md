# Evo Murder Game 项目重构方案

> 目标读者：产品负责人、后端开发、前端开发、数据库开发、AI/LLM 工程师。  
> 当前项目：`evo-murder-game`  
> 重构目标：让项目从原型型工程升级为架构清晰、模块明确、可持续开发、可脱离 EvoMap 独立运行的 AI 剧本杀系统。

---

## 1. 重构结论

当前项目已经具备较完整的产品闭环：剧本库、选角、游戏会话、AI 角色对话、证物系统、投票推理、真相揭示、复盘评分、Gene/Capsule 经验沉淀。真正的问题不是功能少，而是工程边界不清、核心逻辑集中、中文内容乱码、数据库表设计偏原型化、EvoMap 概念渗透过深。

本次重构建议采用“渐进式重构”，不要直接推倒重写。优先目标如下：

1. 修复中文乱码，统一 UTF-8。
2. 拆分后端巨型模块，建立清晰的 service/repository/domain 分层。
3. 重构数据库表，把会话、阶段、对话、证物、Agent、复盘、Skill 资产拆清楚。
4. 将 EvoMap 从核心依赖降级为可选集成。
5. 用本地 Skill 系统替代现有 Capsule 系统。
6. 前端以服务端 snapshot 为唯一局内状态来源，拆分巨型 `GamePage`。
7. 建立基础测试、迁移脚本、API 文档和开发规范。

---

## 2. 当前主要问题

### 2.1 中文乱码

当前 README、后端源码注释、prompt、错误信息、默认值中大量中文已经乱码，例如类似 `鍓ф湰`、`鑷繘鍖` 的文本。该问题不只是终端显示异常，部分乱码已经进入源码字符串，直接影响：

- 前端展示文案
- 后端错误信息
- LLM prompt 质量
- 默认数据库内容
- README 和协作文档可信度

重构前必须先处理编码问题，否则后续 prompt 和文案维护会非常痛苦。

### 2.2 后端职责混杂

当前核心逻辑集中在几个大文件中：

- `api/agents/game_engine.py`：游戏状态机、Agent 局内状态、阶段推进、发言轮、私聊、证物、数据库同步、复盘触发混在一起。
- `api/routes/game.py`：路由层包含大量业务逻辑。
- `api/db/models.py`：所有 ORM 模型和转换函数集中在一个文件。

这会导致新开发者难以定位功能，修改一个功能容易影响其他模块。

### 2.3 前后端状态重复

前端 `GamePage.tsx` 自己维护了大量游戏阶段、发言队列、证物、私聊、事件流状态；后端也维护了 phase、snapshot、agent state、progress 等状态。长期看会导致状态不一致。

重构后应明确：

- 服务端是游戏主状态唯一来源。
- 前端只做交互和展示。
- 阶段推进、投票、证物公开、复盘触发都由后端状态机决定。
- 前端刷新后通过 snapshot 恢复完整 UI。

### 2.4 数据库表偏原型化

当前表虽然覆盖很多数据，但存在几个问题：

- `game_sessions.result` 承载过多 JSON 状态。
- Agent 局内状态、对话、证物、阶段历史边界不够清晰。
- Gene/Capsule 依赖 Agent/EvoMap 命名，资产系统不独立。
- 缺少正式迁移体系，依赖运行时补列。
- 缺少统一审计字段、版本字段和状态字段。

### 2.5 EvoMap 绑定过深

当前项目名称、Agent 注册、Memory、Capsule、Council、A2A Client 等概念都和 EvoMap 关联较深。但从产品角度看，核心玩法并不必须依赖 EvoMap。

建议重构为：

- 核心游戏系统完全本地可运行。
- LLM provider 可配置。
- Agent Runtime 本地化。
- EvoMap 作为可选 adapter，不参与核心领域模型。
- Capsule 改造成本地 Skill 资产。

## 3. 重构后的系统定位

重构后项目应定位为：

> 一个本地优先、可扩展、可接入多种 LLM provider 的 AI 剧本杀引擎。系统支持 AI 角色扮演、证物推理、多人/单人流程、局后复盘，并通过 Skill 资产沉淀可复用的主持、扮演、推理、提示和复盘经验。

核心特点：

- 不依赖 EvoMap 也能完整运行。
- 可选接入 EvoMap、第三方 Agent 平台或外部 Skill 市场。
- Skill 是项目内部可管理、可搜索、可注入 prompt 的资产包。
- 数据库模型清晰，支持迁移和扩展。
- 前后端职责明确。

## 5. EvoMap 解耦方案

### 5.1 结论

项目可以和 EvoMap 解绑，而且建议解绑。

EvoMap 当前在项目中承担了三类概念：

1. Agent 网络注册和 A2A 通信。
2. Memory 经验记录和召回。
3. Capsule/Gene 经验沉淀。

但从现有项目实现看，核心游戏能力并不依赖 EvoMap。多数能力可以本地化：

- Agent 注册可以改成本地 `agents` 表。
- Memory 可以改成本地 `agent_memories` 或 `agent_runtime_states`。
- Capsule 可以改成本地 `skills`。
- A2A/Council 可以作为后续可选集成，不进入主流程。

### 5.2 解耦原则

1. 核心游戏不能依赖 EvoMap 服务可用性。
2. 所有 EvoMap 字段从核心表中移除或改为 optional external reference。
3. 后端不再以 `node_id`、`node_secret` 作为核心 Agent 身份。
4. EvoMap 相关代码移入 `integrations/evomap/`。
5. 所有业务服务依赖抽象接口，不直接依赖 EvoMapClient。

### 5.3 新架构位置

建议：

```text
api/
  integrations/
    evomap/
      client.py
      adapter.py
      schemas.py
  services/
    agent_runtime_service.py
    skill_service.py
    memory_service.py
```

核心服务只依赖：

```python
class AgentMemoryProvider:
    def record(...): ...
    def recall(...): ...

class SkillProvider:
    def search(...): ...
    def load(...): ...
```

默认实现使用本地数据库。EvoMap 实现作为可选 adapter。

### 5.4 字段替换建议

| 当前 EvoMap/Capsule 概念 | 重构后概念 | 说明 |
| --- | --- | --- |
| `node_id` | `agent.id` / `external_ref` | 本地 ID 为主，外部平台 ID 可选 |
| `node_secret` | `external_credentials` | 如需保留，单独加密存储 |
| `memory_record` | `agent_memories` | 本地记忆表 |
| `memory_recall` | `MemoryService.search` | 本地召回 |
| `Gene` | `experience_records` | 原始经验记录 |
| `Capsule` | `skills` | 可复用经验资产 |
| `publish/fetch` | `Skill import/export` | 本地包导入导出 |
| `Council` | 可选集成 | 暂不进入核心流程 |

## 6. Skill 替代 Capsule 方案

### 6.1 为什么用 Skill 替代 Capsule

现有 Capsule 本质上是“从局后经验中提炼出的可复用 prompt 资产”。它已经非常接近 Skill：

- 有标题
- 有适用角色
- 有分类
- 有信号 tags
- 有内容
- 有策略
- 有示例
- 有反模式
- 有评分和使用次数

因此建议不是删除 Capsule 能力，而是将其升级为更加清晰、可文件化、可版本化、可导入导出的 Skill 系统。

### 6.2 Skill 的定位

Skill 是一份可复用的经验/能力包，可以被注入到 Agent 的 system prompt 或运行上下文中。

Skill 可以服务于：

- DM 控场
- AI 角色扮演
- 防剧透
- 证物反应
- 提示生成
- 复盘评分
- 玩家画像
- 剧本推荐
- 推理辅助

### 6.3 Skill 文件包格式

建议每个 Skill 是一个文件夹：

```text
skills/
  dm-pacing-basic/
    skill.json
    prompt.md
    examples.md
    anti-patterns.md
    README.md
```

`skill.json` 示例：

```json
{
  "id": "skill_dm_pacing_basic",
  "name": "DM 控场节奏基础",
  "version": "1.0.0",
  "type": "prompt_skill",
  "category": "hosting",
  "applicableRoles": ["dm"],
  "signals": ["hosting", "pacing", "hint-control"],
  "description": "帮助 DM 在调查阶段控制节奏，避免过早剧透，同时保证玩家不会卡死。",
  "source": {
    "type": "generated_from_experience",
    "experienceId": "exp_xxxxxxxx",
    "sessionId": "session_xxxxxxxx",
    "scriptId": "script_xxxxxxxx"
  },
  "quality": {
    "score": 0.82,
    "reviewStatus": "approved",
    "reviewedBy": "dm",
    "reviewComment": "策略可复用，适合多数推理本。"
  },
  "injection": {
    "mode": "append_system_prompt",
    "priority": 50,
    "maxTokens": 800
  },
  "usage": {
    "usageCount": 0,
    "successCount": 0,
    "lastUsedAt": null
  },
  "createdAt": "2026-06-25T00:00:00Z",
  "updatedAt": "2026-06-25T00:00:00Z"
}
```

`prompt.md` 示例：

```md
## DM 控场节奏基础

当玩家处于调查阶段时，你需要根据当前进度控制信息释放：

1. 玩家没有方向时，只提醒调查目标，不直接给答案。
2. 玩家遗漏关键证物时，优先提示地点或人物关系。
3. 玩家已经接近真相时，可以用反问推动其自己说出结论。
4. 不要主动暴露凶手、作案手法或关键时间线。
```

`examples.md` 示例：

```md
### 适用场景

- 玩家连续多轮没有发现新线索。
- 玩家已经发现证物，但没有建立人物关系。
- 玩家怀疑方向正确，但证据链不完整。

### 示例表达

“你们似乎已经注意到时间线的问题，不妨再看看谁最有机会接触那件物品。”
```

`anti-patterns.md` 示例：

```md
### 不要这样做

- 直接说出凶手。
- 直接解释完整作案手法。
- 在玩家没有搜证前提前公开关键证物。
- 用上帝视角纠正玩家所有错误。
```

### 6.4 Skill 数据库存储

文件包适合导入导出，数据库适合检索和运行时注入。建议二者同时支持：

- `skills/` 目录：保存可导入导出的 Skill 包。
- 数据库 `skills` 表：保存索引、内容、评分、状态、使用统计。

### 6.5 Capsule 到 Skill 的字段映射

| Capsule 字段 | Skill 字段 | 说明 |
| --- | --- | --- |
| `id` | `id` | 保留或重新生成 |
| `gene_id` | `source_experience_id` | Gene 改名 Experience |
| `publisher_id` | `created_by_agent_id` | 发布者变为创建者 |
| `publisher_role` | `applicable_roles` / `created_by_role` | 角色信息拆开 |
| `title` | `name` | 标题 |
| `category` | `category` | 分类 |
| `signals` | `signals` | 检索信号 |
| `applicable_roles` | `applicable_roles` | 适用角色 |
| `content` | `prompt_content` | 主 prompt 内容 |
| `strategy` | `strategy` | 策略 |
| `examples` | `examples` | 示例 |
| `anti_patterns` | `anti_patterns` | 反模式 |
| `score` | `quality_score` | 质量评分 |
| `usage_count` | `usage_count` | 使用次数 |
| `effectiveness` | `effectiveness_score` | 效果评分 |
| `review_status` | `review_status` | 审核状态 |

### 6.6 Skill 注入流程

新局开始或 Agent 回复前：

1. 根据 Agent role、剧本类型、当前阶段、signals 搜索 Skill。
2. 按评分、适用角色、使用效果排序。
3. 控制 token 总量。
4. 将 Skill 内容注入 Agent system prompt。
5. 记录 Skill 使用日志。
6. 局后根据表现更新 Skill 效果评分。

建议接口：

```text
GET    /skills
GET    /skills/{skill_id}
POST   /skills
PATCH  /skills/{skill_id}
DELETE /skills/{skill_id}
POST   /skills/search
POST   /skills/import
GET    /skills/{skill_id}/export
POST   /skills/{skill_id}/review
POST   /sessions/{session_id}/skills/inject
POST   /sessions/{session_id}/skills/feedback
```

## 8. 数据库重构方案

### 8.1 数据库重构目标

数据库重构的目标是把“运行时状态”和“基础配置资产”拆开，把“大 JSON 状态”拆成可查询、可迁移、可维护的结构化表。

原则：

1. 核心业务字段结构化。
2. 不把所有运行时状态塞进 `result` JSON。
3. JSON 字段只保存灵活扩展内容。
4. 所有表统一 `created_at`、`updated_at`。
5. 关键表统一 `status`、`version`。
6. 引入 Alembic 管理迁移。
7. 本地 SQLite 和生产 PostgreSQL 都能工作。

### 8.5 迁移策略

数据库迁移建议分三步走。

#### 第一步：双写兼容

新增新表，但旧表不删。写入时同时写旧表和新表。

目标：降低风险，保证旧前端和旧 API 还能跑。

#### 第二步：读新表，旧表兜底

Service 优先读新表；如果没有数据，再从旧表读取并即时迁移。

目标：逐步完成数据迁移。

#### 第三步：移除旧表依赖

所有 API 和前端都切到新数据结构后，归档旧表。

建议旧表不要直接删除，先改名备份：

```text
legacy_game_sessions
legacy_agent_nodes
legacy_genes
legacy_capsules
```

## 15. 开发纪律

1. 不要边重构边随意加新功能。
2. 所有新业务逻辑必须进入 service。
3. 所有数据库访问必须进入 repository 或 service。
4. 所有 prompt 必须文件化，不再散落在代码中。
5. 所有游戏主状态以后端 snapshot 为准。
6. 所有 schema 变化必须走 Alembic。
7. EvoMap 只能作为可选 integration。
8. Skill 替代 Capsule 后，前端和后端命名都应统一。
9. 保留旧 API 兼容期，但必须设定删除时间。
10. 每拆一个模块，必须补对应测试。

## 16. 最终建议

本项目不建议直接推倒重写。最合理的路线是：

1. 先修乱码和环境。
2. 再重构数据库和后端边界。
3. 然后解耦 EvoMap，把 Capsule 升级为 Skill。
